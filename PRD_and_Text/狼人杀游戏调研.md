# 狼人杀游戏调研

> 调研目的：梳理狼人杀的标准规则体系、核心角色机制、胜负逻辑及常见变体，为 AI 自迭代规则系统提供游戏设计基准。

---

## 一、游戏概述

**狼人杀**（Werewolf / Mafia）起源于1986年苏联心理学家 Dmitry Davidoff 设计的"黑手党"游戏，后经欧美桌游社区改良，加入"狼人"主题和各种神职角色，成为当今最流行的推理社交游戏之一。

### 核心玩法
- 玩家分为**狼人阵营**（少数，知道彼此身份）和**好人阵营**（多数，身份互相不知）
- 游戏在"夜晚"和"白天"之间交替进行
- **夜晚**：狼人秘密击杀一名好人；各神职悄悄使用技能
- **白天**：所有玩家公开讨论，投票放逐一名"疑似狼人"的玩家
- 循环直到一方达成胜利条件

---

## 二、标准角色体系

### 2.1 狼人阵营

| 角色 | 核心能力 | 战略价值 |
|------|---------|---------|
| **狼人（Werewolf）** | 夜晚与队友共同击杀一名好人；知道所有狼人队友身份 | 伪装好人，引导投票方向，保护队友 |
| **狼王/白狼王** | 被投票出局时可开枪带走一人（变体） | 出局不吃亏，威慑好人随意投票 |
| **恶狼/毒狼** | 击杀好人时可附带特殊效果（变体） | 克制女巫等神职 |

### 2.2 好人阵营 — 神职

| 角色 | 核心能力 | 战略价值 |
|------|---------|---------|
| **预言家（Seer）** | 每夜查验一名玩家身份（好人/狼人） | 信息核心，需谨慎发言时机以防被针对 |
| **女巫（Witch）** | 持有解药（救人）和毒药（杀人）各一瓶 | 关键时刻改变局势，药水时机至关重要 |
| **猎人（Hunter）** | 出局时（被杀或被投票）可开枪带走一名玩家 | 出局代价，威慑狼人随意击杀 |
| **守卫（Guard）** | 每夜守护一名玩家，被守护者当夜免死（变体） | 保护高价值目标，但不能连续守同一人 |
| **村民（Villager）** | 无特殊技能，只能通过推理参与白天投票 | 信息搜集与投票主体 |

### 2.3 第三方阵营（变体）

| 角色 | 胜利条件 | 说明 |
|------|---------|------|
| **丘比特** | 两名"恋人"共存活或同时出局 | 可能导致好人与狼人结盟 |
| **混沌角色** | 个人独立胜利条件 | 增加不确定性 |

## 三、胜负条件

### 标准胜负判定

| 阵营 | 胜利条件 |
|------|---------|
| **狼人胜** | 存活狼人数量 ≥ 存活好人数量（经典条件） |
| **村民胜** | 所有狼人全部出局 |
| **平局** | 游戏超过最大天数限制（通常12天）未分出胜负 |

### 胜负平衡的核心矛盾
- **信息不对称**：狼人知道所有人身份，好人只能推理
- **人数差异**：狼人少但信息优势大；好人多但信息盲区大
- **资源消耗**：双方每轮各损失1人，但损失的关键性不同

## 四、标准游戏流程

### 游戏前
1. 主持人分发角色牌（秘密分配）
2. 玩家确认自己的角色，记住身份

### 第一夜（特殊）
```
主持人：天黑请闭眼
  → 狼人睁眼，互相确认队友身份，选择击杀目标
  → 狼人闭眼
  → 预言家睁眼，查验一名玩家（主持人用手势示意好人/狼人）
  → 预言家闭眼
  → 女巫睁眼，主持人告知今晚被杀者，女巫决定是否救人/是否毒人
  → 女巫闭眼
主持人：天亮了
```

### 白天流程
```
主持人宣布：昨夜发生的事（谁死了 / 平安夜）
  → 遗言（被杀者可遗言，具体规则因版本而异）
  → 自由讨论（所有存活玩家按顺序或自由发言）
  → 投票：每人投票放逐一名玩家
  → 宣布结果：得票最多者出局（平票规则视版本）
  → 出局者遗言 / 特殊技能触发（猎人开枪等）
检查胜负条件
```

### 后续夜晚
与第一夜流程相同，但：
- 若预言家已出局，跳过预言家环节
- 若女巫药水已用完，跳过女巫环节

## 五、核心策略与博弈

### 5.1 狼人阵营策略
- **混水摸鱼**：制造好人之间的争论，转移怀疑
- **跟票**：跟随多数投票方向，避免暴露
- **反咬**：主动指控威胁最大的好人（尤其是预言家）
- **保狼**：为队友辩护，但不能过于明显
- **刀神**：优先击杀神职（预言家>女巫>猎人）

### 5.2 好人阵营策略
- **预言家发言**：何时公开身份、公开查验结果，是核心决策
- **女巫时机**：解药救谁、毒药毒谁，往往决定胜负
- **逻辑归票**：通过发言逻辑分析谁可能是狼人
- **警惕"跟刀"**：连续两天被投票高票者可能被针对

### 5.3 信息博弈核心
- 好人的核心资产：预言家的查验结果
- 狼人的核心目标：让好人不相信真预言家，或尽早击杀预言家
- 制造"假预言家"（狼人伪装预言家）是常见战术

## 六、平衡性分析

### 6.1 影响平衡的核心变量

| 变量 | 向狼人倾斜 | 向好人倾斜 |
|------|----------|----------|
| 狼人数量 | 更多狼人 | 更少狼人 |
| 神职存在 | 无神职 | 有预言家+女巫+猎人 |
| 女巫自救 | 不允许 | 允许 |
| 第一夜免死 | 否 | 是 |
| 猎人机制 | 限制开枪 | 扩大开枪条件 |
| 胜利条件 | 狼人≥好人即胜 | 要求更高才算狼人胜 |
| 警长机制 | 无 | 有（好人归票更有效率） |

### 6.2 标准8人局平衡基准

```
标准配置：2狼 + 预言家 + 女巫 + 猎人 + 3村民
理论胜率：好人略占优（约55-60%）
原因：两个强力神职（预言+女巫）信息量大于2狼优势
```

### 6.3 常见失衡原因
- **狼人过多**（如3狼/8人）→ 狼人胜率显著上升
- **关键神职被第一夜击杀** → 剩余好人缺乏信息来源
- **女巫解药误用** → 救错人导致好人阵营减员加速
- **好人误票** → 连续放逐好人，狼人获胜

## 七、常见规则变体

### 7.1 警长制度
- 白天开始前先竞选警长，警长票数算1.5票
- **影响**：大幅增强好人归票效率；竞选环节增加博弈；狼人会争抢警长以误导投票
- **平衡评估**：整体有利于好人，尤其是多狼局

### 7.2 第一夜免死
- 第一夜没有击杀，天亮后所有人存活
- **影响**：给预言家更多时间查验；好人获得额外讨论机会
- **平衡评估**：有利于好人

### 7.3 女巫不可自救
- 女巫若自己被狼人击杀，无法使用解药救自己
- **影响**：削弱女巫生存能力，加快神职消耗
- **平衡评估**：有利于狼人

### 7.4 女巫同夜双药
- 女巫可在同一夜既使用解药又使用毒药
- **影响**：大幅增强女巫单夜影响力，可能一夜改变胜负
- **平衡评估**：整体有利于好人，但波动性大

### 7.5 猎人扩展开枪条件
- 猎人被投票放逐时也可开枪（标准规则仅夜晚被杀时可开枪）
- **影响**：好人敢于放逐疑似猎人的玩家；狼人更谨慎
- **平衡评估**：微利于好人

### 7.6 狼人数量变化
| 配置 | 8人局 | 胜率影响 |
|------|-------|---------|
| 1狼 | 1/8 | 好人大幅优势 |
| 2狼（标准） | 2/8 | 基本平衡（好人略占优） |
| 3狼 | 3/8 | 狼人占优 |

## 八、与"规则自迭代系统"的结合点

### 8.1 哪些规则适合被玩家投票修改
- **平衡类**：狼人数量、胜利条件 → 直接影响胜率，争议最大
- **机制类**：女巫自救、猎人开枪条件 → 改变神职价值，讨论空间大
- **节奏类**：第一夜免死、警长机制 → 影响游戏节奏和信息密度
- **不适合修改**：游戏基本结构（夜晚/白天交替、投票放逐机制）

### 8.2 规则演化的预期动态

```
初始状态（标准规则，好人略占优）
    ↓
狼人友好玩家（阿狼）持续推动强化狼人的规则
    ↓
好人友好玩家（守护者）反向推动保护神职的规则
    ↓
平衡派玩家调节，分析师提供数据支撑
    ↓
规则在博弈中动态演化，形成独特的"本桌规则集"
```

### 8.3 规则变化的监控指标
- **胜率**：狼人/村民各轮胜率（目标维持在40-60%区间）
- **游戏时长**：平均持续天数（过短说明一方过强）
- **规则触发率**：每条规则实际被触发的频率（未触发的规则可能无效）
- **讨论热度**：哪条规则在讨论阶段被最多提及

## 九、Agent 玩狼人杀的框架设计 — 开源项目调研

> 调研时间：2026年3月。梳理当前 LLM Agent 玩狼人杀/Mafia 游戏的主要开源项目和学术框架，为本项目的架构优化提供参考。

### 9.1 通用多智能体游戏平台

#### ChatArena（Farama Foundation）
- **链接**：https://github.com/Farama-Foundation/chatarena
- **星标**：1.5k ⭐ | **状态**：已停止维护（2025年8月宣布废弃）
- **框架思路**：
  - 基于 **马尔科夫决策过程（MDP）** 抽象多智能体语言游戏环境
  - 四层架构：**Arena**（主循环+HCI）→ **Environment**（游戏状态+规则逻辑）→ **Player**（策略=观察→动作映射）→ **Language Backend**（LLM 推理引擎）
  - 游戏状态对玩家**不可见**，玩家只能看到自然语言形式的 observation
  - 内置环境包括 Conversation、Moderator Conversation、**Chameleon**（社交推理游戏，含线索→投票→猜测三阶段）
  - 支持 Gradio WebUI 和 CLI 两种交互方式
- **对本项目的参考价值**：
  - Environment 抽象（状态转移 + observation 渲染）可参考用于解耦游戏逻辑和 AI 交互
  - Chameleon 环境的"投票 + 社交推理"机制是狼人杀的简化版，架构可直接借鉴
  - 已废弃，不建议直接依赖，但架构设计理念依然有价值

#### AgentScope（阿里 ModelScope）
- **链接**：https://github.com/agentscope-ai/agentscope
- **星标**：21.1k ⭐ | **状态**：活跃维护
- **框架思路**：
  - 生产级多智能体框架，核心理念是**利用模型的推理和工具使用能力**而非用严格 prompt 约束
  - 核心组件：**ReActAgent**（推理+行动）、**MsgHub**（消息广播中心，管理多智能体对话）、**Pipeline**（编排顺序/并发执行流）
  - 内置 **九人狼人杀示例**（`examples/game/werewolves`），包含完整游戏流程
  - 支持 **Agentic RL 训练**：可用强化学习微调狼人杀 Agent（示例数据：Qwen2.5-7B 狼人胜率从50%→80%）
  - 支持语音 Agent、MCP 协议、A2A（Agent-to-Agent）协议
- **对本项目的参考价值**：
  - MsgHub 的消息广播机制可参考用于我们的规则讨论阶段（多人同时收听、轮流发言）
  - Agentic RL 训练思路：未来可考虑用 RL 优化玩家 Agent 的策略
  - 九人狼人杀示例可直接参考其角色提示词设计和游戏流程编排

### 9.2 学术研究项目

#### KylJin/Werewolf — RL-instructed LLM Agent（NeurIPS 2024）
- **链接**：https://github.com/KylJin/Werewolf
- **论文**：*Learning to Discuss Strategically: A Case Study on One Night Ultimate Werewolf*（NeurIPS 2024）
- **星标**：10 ⭐
- **框架思路**：
  - 三组件架构：
    1. **Belief Modeling（信念建模）**：基于观察形成对其他玩家角色的信念分布
    2. **Discussion Tactic Selection（讨论策略选择）**：用 **RL 训练的策略网络** 从候选讨论策略中选择最优策略
    3. **Decision Making（决策）**：结合观察、信念和讨论策略做出行动决策
  - 游戏环境基于 **One Night Ultimate Werewolf（ONUW）** 变体（单夜制：夜晚→讨论→投票，一轮定胜负）
  - LLM 负责语言生成，RL 负责**战术层面的策略选择**，实现"RL 指导 LLM"的混合架构
- **对本项目的参考价值**：
  - **信念建模**思路：让 Agent 显式维护对其他玩家身份的概率估计，而非仅靠 prompt
  - **RL + LLM 混合架构**：策略层用 RL 优化、表达层用 LLM 生成，避免 LLM 在策略推理上的弱点
  - 我们的规则讨论阶段也可以引入类似的"策略选择"机制（支持/反对/提修改/保持沉默）

#### niveck/LLMafia — 异步通信 Agent（EMNLP 2025 Findings）
- **链接**：https://github.com/niveck/LLMafia
- **论文**：*Time to Talk: LLM Agents for Asynchronous Group Communication in Mafia Games*（EMNLP 2025 Findings）
- **星标**：22 ⭐
- **框架思路**：
  - 核心创新：解决 LLM 在**异步多人对话**中"何时说话"的问题
  - 双模块架构：
    1. **Scheduler（调度器）**：决定"何时发言"——持续模拟是否应当在当前时刻发送消息
    2. **Generator（生成器）**：决定"说什么"——生成消息内容
  - LLM Agent 与**真人玩家**混合对局，在异步群聊环境中玩 Mafia 游戏
  - 提供 HuggingFace 数据集（所有对局的完整消息记录）
  - 实验结果：Agent 在游戏表现和融入度方面与真人玩家持平
- **对本项目的参考价值**：
  - Scheduler 模块的设计思路：我们的规则讨论阶段是否也需要 Agent 自主判断"是否发言"而非强制轮流
  - 人机混合对局的经验：未来可扩展为支持真人玩家参与

#### ReneeYe/MaKTO-Werewolf — 多智能体 KTO 训练
- **链接**：https://github.com/ReneeYe/MaKTO-Werewolf
- **论文**：*Multi-agent KTO: Reinforcing Strategic Interactions of Large Language Model in Language Game*
- **星标**：3 ⭐
- **框架思路**：
  - 使用 **KTO（Kahneman-Tversky Optimization）** 替代传统 RLHF 来训练狼人杀 Agent
  - 多智能体交互中的策略强化：不仅优化单个 Agent，而是在多 Agent 博弈过程中联合优化
  - 9人制狼人杀，含完整角色体系
- **对本项目的参考价值**：
  - KTO 训练方法可作为未来 Agent 能力提升的技术储备
  - 多智能体联合训练思路有助于规避"所有 Agent 策略趋同"的问题

### 9.4 框架设计模式总结

基于以上调研，Agent 玩狼人杀的框架设计普遍遵循以下模式：

```
┌─────────────────────────────────────────────────┐
│                  Game Engine                     │
│  (状态机：管理阶段转换、胜负判定、回合计数)          │
└──────────────┬─────────────┬────────────────────┘
               │             │
    ┌──────────▼──────┐ ┌───▼──────────────┐
    │   Environment   │ │   Message Hub    │
    │ (游戏状态/规则)  │ │ (消息广播/路由)   │
    └──────────┬──────┘ └───┬──────────────┘
               │             │
    ┌──────────▼─────────────▼────────────────┐
    │              Agent Layer                 │
    │  ┌─────────┐  ┌─────────┐  ┌─────────┐ │
    │  │ Belief  │→ │Strategy │→ │Generator│ │
    │  │ Module  │  │ Selector│  │ (LLM)   │ │
    │  └─────────┘  └─────────┘  └─────────┘ │
    └─────────────────────────────────────────┘
```

**通用设计原则**：
1. **状态与观察分离**：Agent 不能直接访问游戏状态，只能通过 observation 获取信息
2. **角色提示词模板化**：每个角色有独立的系统提示词，定义能力、目标和行为约束
3. **消息路由控制**：不同阶段（白天公开/夜晚私密）的消息可见范围不同
4. **记忆管理**：Agent 维护历史对话记录和关键事件摘要，避免上下文超限
5. **混合决策架构**：策略层（RL/规则）+ 表达层（LLM）分离，提高决策质量

### 10.2 消息管理：MsgHub 机制

AgentScope 的 **MsgHub** 是核心消息管理原语，通过上下文管理器(`async with`)控制消息的广播范围和自动广播行为。

#### MsgHub 的三种使用模式

| 模式 | 代码示例 | 场景 |
|------|---------|------|
| **全体广播** | `MsgHub(participants=agents)` | 游戏开始时角色分配通知 |
| **分组私聊（自动广播）** | `MsgHub(players.werewolves, enable_auto_broadcast=True)` | 狼人夜间讨论 |
| **手动广播** | `MsgHub(participants, enable_auto_broadcast=False)` | 投票环节，防止互相影响 |

#### 关键设计细节

```python
# 1. 全体 MsgHub（手动广播，精确控制每条消息的广播时机）
async with MsgHub(
    participants=players.current_alive,
    enable_auto_broadcast=False,
    name="alive_players",
) as alive_players_hub:
    await alive_players_hub.broadcast(await moderator(Prompts.to_all_night))

# 2. 狼人讨论 MsgHub（嵌套在全体 hub 内，自动广播 + announcement 初始消息）
async with MsgHub(
    players.werewolves,
    enable_auto_broadcast=True,
    announcement=await moderator(Prompts.to_wolves_discussion.format(...)),
    name="werewolves",
) as werewolves_hub:
    # 讨论时自动广播每条发言
    for _ in range(1, MAX_DISCUSSION_ROUND * n_werewolves + 1):
        res = await players.werewolves[_ % n_werewolves](
            structured_model=DiscussionModel,
        )
        if _ % n_werewolves == 0 and res.metadata.get("reach_agreement"):
            break
    # 投票时关闭自动广播
    werewolves_hub.set_auto_broadcast(False)
    msgs_vote = await fanout_pipeline(players.werewolves, ...)
```

**核心洞察**：
- MsgHub 支持**动态切换** `set_auto_broadcast(True/False)`，在同一个 hub 内实现"讨论可见→投票不可见"
- MsgHub 支持**嵌套**：狼人私聊 hub 嵌套在全体 hub 内，离开狼人 hub 后消息不会泄露给好人
- `announcement` 参数在创建 hub 时自动发送初始消息给所有参与者
- MsgHub 内部维护 `participants` 列表，自动将消息路由给列表中的 agent

### 10.3 流程推进：Pipeline 机制

AgentScope 使用两种 pipeline 控制 agent 调用顺序：

| Pipeline | 功能 | 用途 |
|----------|------|------|
| `sequential_pipeline` | 按顺序依次调用每个 agent，前一个的输出作为后一个的输入 | **白天讨论**：玩家轮流发言 |
| `fanout_pipeline` | 并行调用所有 agent（同一条消息），收集所有响应 | **投票环节**：所有人同时投票，互不影响 |

#### 白天讨论流程

```python
# 开启自动广播 → sequential_pipeline 让每人轮流说话 → 关闭自动广播
alive_players_hub.set_auto_broadcast(True)
await sequential_pipeline(players.current_alive)
alive_players_hub.set_auto_broadcast(False)
```

#### 白天投票流程

```python
# fanout_pipeline 并行投票，enable_gather=False 表示不汇总消息
msgs_vote = await fanout_pipeline(
    players.current_alive,
    await moderator(Prompts.to_all_vote.format(names_to_str(...))),
    structured_model=get_vote_model(players.current_alive),
    enable_gather=False,
)
voted_player, votes = majority_vote([_.metadata.get("vote") for _ in msgs_vote])
# 投票后统一广播结果，避免先投的人影响后投的人
await alive_players_hub.broadcast([*msgs_vote, await moderator(Prompts.to_all_res.format(...))])
```

#### 完整游戏流程状态机

```
游戏开始 → 角色分配
    ↓
┌→ 夜晚阶段
│   ├─ 狼人讨论（MsgHub 嵌套，auto_broadcast=True）
│   ├─ 狼人投票（fanout_pipeline，auto_broadcast=False）
│   ├─ 女巫行动（私密，observe 接收结果）
│   ├─ 预言家行动（私密）
│   └─ 猎人判定（被杀且非毒杀时触发）
│   ↓
│  更新存活 → 胜负判定
│   ↓
│  白天阶段
│   ├─ 死亡公告
│   ├─ 遗言环节（first_day 限定）
│   ├─ 讨论（sequential_pipeline，auto_broadcast=True）
│   ├─ 投票（fanout_pipeline，auto_broadcast=False，统一广播）
│   ├─ 投票出局者遗言
│   └─ 猎人判定（被投票出局时触发）
│   ↓
│  更新存活 → 胜负判定
│   ↓
└─ 下一轮（first_day = False）

游戏结束 → 每人反思（fanout_pipeline，仅自己可见）
```

### 10.4 记忆与历史管理

#### Agent 层面：ReActAgent 内置记忆

- 每个 `ReActAgent` 内置 message history，所有 `observe()` 和自身回复自动记录
- MsgHub 广播的消息自动调用每个参与者的 `observe()`，等效于"听到了"
- 系统提示词在 `main.py` 中一次性设定，包含**角色规则、策略指导、NOTE 约束**

#### Session 层面：JSONSession 持久化

```python
# 加载上一局的 agent 状态（记忆、经验）
session = JSONSession(save_dir="./checkpoints")
await session.load_session_state(
    session_id="players_checkpoint",
    **{player.name: player for player in players},
)
# 游戏结束后保存
await session.save_session_state(session_id="players_checkpoint", ...)
```

**连续对局价值**：Agent 可以在多局游戏中积累经验，学习和优化策略。

#### 反思机制

游戏结束后，所有 agent 进行自我反思：
```python
await fanout_pipeline(
    agents=agents,
    msg=await moderator(Prompts.to_all_reflect),
)
```
反思内容仅自己可见（fanout 不在 MsgHub 内），写入 agent 记忆，影响下一局策略。

### 10.6 结构化输出模型（structured_model.py）

AgentScope 使用 **Pydantic BaseModel** 控制 LLM 输出格式：

| 模型 | 字段 | 用途 |
|------|------|------|
| `DiscussionModel` | `reach_agreement: bool` | 狼人讨论是否达成一致 |
| `VoteModel` | `vote: Literal[玩家名]` | 投票选择，用 Literal 约束为存活玩家 |
| `WitchResurrectModel` | `resurrect: bool` | 女巫是否救人 |
| `WitchPoisonModel` | `poison: bool, name: Literal[玩家名]` | 女巫是否毒杀及目标 |
| `SeerModel` | `name: Literal[玩家名]` | 预言家查验目标 |
| `HunterModel` | `shoot: bool, name: Literal[玩家名]` | 猎人是否开枪及目标 |

**关键设计**：
- `get_vote_model(players.current_alive)` 动态生成模型类，`Literal` 类型约束为当前存活玩家名
- `model_validator` 处理逻辑依赖（如 `poison=False` 时自动清空 `name`）
- 输出存储在 `msg.metadata` 字典中，供游戏引擎读取决策结果

### 10.7 工具类设计（utils.py）

| 组件 | 功能 |
|------|------|
| `Players` 类 | 维护玩家状态：按角色分组、存活列表、名称→角色/agent 映射、胜负判定 |
| `EchoAgent` | 主持人 agent，不调用 LLM，只转发消息（加上 Moderator 身份） |
| `majority_vote()` | 多数投票统计，返回得票最高者和投票分布字符串 |
| `names_to_str()` | 将 agent 列表格式化为 "A, B, and C" 形式 |

### 10.8 对本项目的启示

| AgentScope 特性 | 本项目可借鉴点 |
|-----------------|---------------|
| MsgHub 嵌套 + 动态 auto_broadcast | 实现讨论可见/投票不可见的精确消息控制 |
| sequential_pipeline / fanout_pipeline | 讨论用顺序流水线，投票用并行流水线 |
| Pydantic 结构化输出 | 用结构化模型约束投票/决策输出，避免 LLM 输出解析失败 |
| JSONSession 持久化 | 多局记忆存储，支持 agent 跨局学习策略 |
| 反思机制 | 游戏结束后 agent 自我反思，提升后续表现 |
| EchoAgent 主持人 | 主持人不需要 LLM 推理，纯转发节省 token |
| Literal 类型动态约束 | 根据存活玩家动态生成合法选项，杜绝无效投票 |
