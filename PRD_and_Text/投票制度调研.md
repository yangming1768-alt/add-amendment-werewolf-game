# 世界议会投票制度调研

> 调研目的：为 AI 狼人杀"规则自迭代"系统的提案→讨论→修改→投票流程提供制度设计参考。

---

## 一、英国威斯敏斯特制度（下议院三读制）

### 核心流程
- **一读**：纯形式宣读，无辩论，自动进入议程
- **二读**：讨论议案**原则和方向**，不涉及细节，通过后进入委员会
- **委员会阶段**：逐条技术性审查，可召开公众听证，接受书面证据
- **报告阶段**：全院审议委员会修改结果，可提出进一步修正案
- **三读**：最终投票，通常仅讨论文本，不再允许实质修正

### 特色机制
| 机制 | 说明 |
|------|------|
| **议长选择权（Speaker's Selection）** | 议长有权筛选哪些修正案付诸表决，防止修正案炸弹 |
| **主题相关性规则（Relevance Rule）** | 修正案必须与议案主题相关，不得借修正偷换议案本质 |
| **阶段隔离原则** | 每阶段任务清晰，不允许跨阶段跳跃 |

### 可借鉴的洞察
- 「讨论原则」和「讨论细节」应该分开，不能混在一轮里
- 修正案需要经过筛选才能进入投票，防止稀释讨论焦点

## 二、美国国会立法流程（委员会中心制）

### 核心流程
1. 议员提案，投入"篓筐"（Hopper）启动流程
2. 议长/多数党领袖将议案**分配给相应委员会**
3. 委员会主席可将议案搁置不审（**Pigeonhole**），约90%的提案死于此处
4. 委员会内部：小组委员会 → **Markup Session（逐条修改）** → 形成委员会报告
5. **规则委员会**规定辩论时长、允许提出的修正案范围
6. 全院辩论 → 投票

### 特色机制
| 机制 | 说明 |
|------|------|
| **委员会死亡权** | 大量低质量提案被过滤，不进入公开辩论 |
| **Markup Session** | 逐行审查文本，精确修改，是真正的技术性修改环节 |
| **封闭/开放规则（Closed/Open Rule）** | 规则委员会可规定是否允许提修正案 |
| **Filibuster（冗长发言）** | 少数派可通过无限期发言阻止表决，须60票才能终止 |
| **Cloture（终辩动议）** | 超级多数（60票/100）才能强行结束讨论 |

### 投票门槛
- 普通法案：简单多数（50%+1）
- 推翻总统否决：两院各2/3多数
- 条约批准：参议院2/3多数
- 宪法修正案：两院各2/3 + 3/4州批准

### 可借鉴的洞察
- 提案进入正式讨论前应先过一道技术性初审
- 不同重要程度的规则修改应对应不同的通过门槛
- 少数派保护机制（Filibuster）有价值，但需要有终止机制，防止永久阻断

## 三、欧洲议会（共同决策程序）

### 特色机制
| 机制 | 说明 |
|------|------|
| **报告员制度（Rapporteur）** | 每个议案指定一名报告员负责推进，来自主责委员会 |
| **影子报告员（Shadow Rapporteur）** | 其他党团各指定一人监督，形成平行审查 |
| **妥协修正案合并** | 相似修正案鼓励合并为一个妥协版本再投票，减少投票轮次 |
| **三方谈判（Trilogue）** | 正式投票前，议会、理事会、委员会三方预协商，减少无效投票 |
| **绝对多数门槛** | 不是"出席者过半"而是"全体议员过半"，门槛更高 |

### 可借鉴的洞察
- "提案方 + 反对方各有代表人"的结构化对抗更公平
- 投票前先协商，减少正式表决失败概率
- 相似提案合并，防止投票疲劳

## 四、古罗马元老院

### 特色机制
| 机制 | 说明 |
|------|------|
| **Quid Censes（强制表态）** | 执政官逐一询问每位元老，每人必须留下立场记录，不能沉默 |
| **等级发言顺序** | 按资历严格排序，前执政官优先，防止争抢发言 |
| **Discessio（分立预投票）** | 赞成者与反对者分立两侧，可视化展示当前分布，再正式计票 |
| **Intercessio（护民官否决）** | 特定角色享有一次性否决权，须当场宣布 |
| **古典Filibuster** | 元老可通过超长发言拖延，无时间限制 |

### 可借鉴的洞察
- 强制每人表态，杜绝沉默搭便车
- 预投票先展示分布，让玩家看到他人倾向后再正式投票，增加博弈深度

## 五、瑞士直接民主制度

### 特色机制
| 机制 | 说明 |
|------|------|
| **联署门槛** | 需一定人数签名才能发起公投（联邦层级需10万人/18个月） |
| **可选性公投（Optional Referendum）** | 议会通过的法律，若5万人联署可要求复决 |
| **强制性公投（Mandatory Referendum）** | 宪法修正案必须公投，无需联署 |
| **双重多数（Double Majority）** | 全国投票者过半 AND 26个州中过半数州同意，防止人口大州压倒小州 |
| **反提案机制（Counter-Proposal）** | 政府可针对公民倡议提出反提案，三选一投票（原案/反案/维持现状） |
| **协商撤案** | 提案方可撤案换取政府做出让步 |

### 可借鉴的洞察
- 三选一投票（原案/反案/维持现状）比简单的是/否投票更能反映真实偏好
- 双重多数：同时满足多个维度的多数才算通过，更稳健
- 允许撤案+谈判，增加流程弹性

## 六、古希腊雅典民主

### 特色机制
| 机制 | 说明 |
|------|------|
| **图拉投票（Graphe Paranomon）** | 任何公民可起诉某项法律"违宪"；若提案被大比例否决，提案人被罚款；三次败诉者永久失去提案权 |
| **陶片放逐（Ostracism）** | 每年一次，公民投票将最危险的人放逐10年，无需具体罪名 |

### 波兰自由否决权（反面教材）
- **Liberum Veto（1652–1791）**：任何一名议员喊"我不同意"即可终止整个议会会期，且所有已通过决议全部作废
- 结果：议会几乎瘫痪，被周边列强利用，最终导致波兰三次瓜分亡国
- **教训**：单人否决权会导致系统瘫痪；少数保护必须有边界

### 可借鉴的洞察
- 提案人应为提案质量负责，烂提案有成本
- 定期"废除最不受欢迎规则"投票，是主动优化规则集的好机制
- 绝不允许单人否决权

## 七、德国联邦议院

### 特色机制
| 机制 | 说明 |
|------|------|
| **建设性不信任投票（Konstruktives Misstrauensvotum）** | 推翻政府的不信任投票必须同时提名新总理，否则无效 |

### 可借鉴的洞察
- **否决必须附带替代方案**：不能只否定，反对一条规则必须同时提出替代规则

## 八、综合设计原则（面向狼人杀规则系统）

### 提案过滤漏斗
```
任意玩家想法
    ↓
个人草案（需满足格式：标题 + 问题描述 + 解决方案 + 预期效果）
    ↓
联署门槛（需 N 名玩家联署支持）
    ↓
初审（AI评估：是否可执行？是否与现有规则冲突？）
    ↓
进入正式讨论队列（限制同时讨论数量）
    ↓
结构化辩论（原则讨论 → 细节讨论）
    ↓
修正案阶段（主题相关性检查 + 相似修正案合并）
    ↓
投票表决（分层门槛）
    ↓
生效 / 否决 / 冷却期
```

### 分层投票门槛建议

| 提案类型 | 通过门槛 | 参考来源 |
|---------|---------|---------|
| 小规则调整（修改数字/参数） | 简单多数 >50% | 英国普通法案 |
| 新增规则 | 绝对多数 ≥60% | 欧洲议会一读 |
| 删除现有规则 | 超级多数 ≥2/3 | 美国推翻否决 |
| 核心机制改变（改胜利条件等） | 3/4多数 + 冷却期 | 瑞士宪法修正 |
| 修改提案流程本身（元规则） | 全体一致或≥4/5 | 最高保护 |

### 少数派保护分层

| 层级 | 机制 | 条件 |
|------|------|------|
| 第一层 | 发言保护：反对方享有等量发言时间 | 无条件 |
| 第二层 | 延迟权：可申请延迟投票一轮 | >25%反对，有限次数 |
| 第三层 | 存档权：反对意见必须被记录在案 | 无条件 |
| 第四层 | 复决权：规则实施后可触发复决 | 需一定比例联署 |
| 底线 | 某些基础规则不可通过任何多数修改 | 宪法地位 |

### 防劣化机制

- **提案成本**：提案被以超过2/3反对票否决，提案人消耗信誉积分（参考雅典图拉投票）
- **定期废除投票**：每N局后投票选出"最不受欢迎的规则"，得票最多者进入废除候选
- **版本回滚**：若新规则导致关键指标（胜率/游戏时长）超出阈值，触发紧急回滚
- **规则健康追踪**：AI追踪每条规则实际触发频率，长期未触发的规则自动进入"休眠审查"

## 九、Agent 投票/辩论/共识框架设计 — 开源项目调研

> 调研时间：2026年3月。梳理当前 LLM 多智能体在投票、辩论、共识决策方面的主要开源项目和学术框架，为规则自迭代系统的提案→讨论→投票流程提供技术参考。

### 9.1 学术研究项目

#### Debate or Vote（Wisconsin 大学，NeurIPS 2025 Spotlight）
- **链接**：https://github.com/deeplearning-wisc/debate-or-vote
- **论文**：*Debate or Vote: Which Yields Better Decisions in Multi-Agent Large Language Models?*（[arxiv.org/abs/2508.17536](https://arxiv.org/abs/2508.17536)）
- **星标**：58 ⭐
- **框架思路**：
  - 系统性比较两种多 Agent 决策策略：**辩论（Debate）** vs **投票（Vote）**
  - **辩论模式**：多个 Agent 进行多轮讨论，逐步修正自己的观点，最终由一个 Agent 或聚合函数给出最终答案
    - **Full MAD（Multi-Agent Debate）**：所有 Agent 相互可见所有发言
    - **Sparse MAD**：Agent 只能看到部分其他 Agent 的发言（模拟信息不对称）
    - **Centralized MAD**：由中央裁判收集各 Agent 意见并汇总
  - **投票模式**：每个 Agent 独立给出答案，通过多数投票决定最终结论
  - **核心发现**：
    - 在简单任务上，投票比辩论更高效（节省 token）
    - 在复杂推理任务上，辩论能通过信息交换提升质量
    - **多人格（Multi-Persona）** 设置（给 Agent 不同性格/角色）会影响辩论质量
  - 支持 5 个 Agent 并行、5 轮辩论/投票的实验设置
- **对本项目的参考价值**：
  - 直接回答了"规则讨论应该用辩论还是投票"的核心设计问题
  - 我们当前的设计（先辩论再投票）有学术依据支持：**辩论+投票的混合模式**是最优的
  - Sparse MAD 的思路可参考：不是所有玩家都需要听到所有发言，可按阵营/资历限制信息可见性
  - Multi-Persona 的研究验证了我们"给 AI 玩家不同性格"的设计方向

#### MALLM — Multi-Agent LLMs for Conversational Task-Solving
- **链接**：https://github.com/Multi-Agent-LLMs/mallm
- **星标**：54 ⭐
- **框架思路**：
  - 通用的多 Agent 对话式任务求解框架
  - 核心架构：**Panelist（小组成员）** → **Discussion Protocol（讨论协议）** → **Decision Aggregation（决策聚合）**
  - 支持多种讨论协议：
    - **Round-Robin（轮流发言）**：每个 Agent 按顺序发言
    - **Open Forum（自由讨论）**：Agent 自行决定是否发言
    - **Structured Debate（结构化辩论）**：指定正方/反方角色
  - 支持多种决策聚合方式：多数投票、加权投票、共识判定
  - 标签：`multi-agent`、`debate`、`multi-agent-debate`
- **对本项目的参考价值**：
  - Discussion Protocol 的抽象设计可直接参考用于规则讨论阶段
  - 多种聚合方式的对比数据可帮助我们选择最优的投票机制
  - 结构化辩论（正方/反方）与我们的提案方/反对方设计高度吻合

#### Agent4Debate（ICASSP 2026）
- **链接**：https://github.com/ZhangYiqun018/agent-for-debate
- **星标**：36 ⭐
- **框架思路**：
  - 动态多智能体框架，目标是让 LLM 达到**人类水平的竞技辩论**能力
  - 多 Agent 角色分工：
    - **Debater（辩手）**：进行正反方辩论
    - **Judge（裁判）**：评估辩论质量
    - **Coach（教练）**：提供策略指导
  - 动态策略调整：Agent 根据对手发言实时调整辩论策略
- **对本项目的参考价值**：
  - 辩论中的"裁判"角色对应我们的主持人（HostAgent）评估提案
  - 动态策略调整的思路：玩家在辩论中可以根据其他人的发言改变立场

#### MAKER Framework — Maximal Agentic Decomposition + Ahead-by-K Voting
- **链接**：https://github.com/rchasman/dominion-maker
- **论文**：arXiv:2511.09030
- **星标**：2 ⭐
- **框架思路**：
  - **Maximal Agentic Decomposition（最大化智能体分解）**：将复杂任务拆解为最小子任务，每个子任务由独立 Agent 执行
  - **Ahead-by-K Voting（超前K票投票）**：不是简单多数投票，而是要求胜方比第二名多 K 票才算通过
  - **Error Correction（纠错）**：投票结果通过后，再经过一轮纠错验证
  - 多个 LLM 模型混合使用（异构 Agent 池）
- **对本项目的参考价值**：
  - Ahead-by-K Voting 机制比简单多数更稳健，可考虑用于核心规则修改（如：不仅需要>50%，还需要比反对票多3票）
  - Error Correction 步骤：规则通过后增加一轮"合理性验证"

### 9.2 社区平台项目

#### Platform War — 多 LLM Agent 社交平台辩论
- **链接**：https://github.com/LYiHub/platform-war-public
- **星标**：786 ⭐
- **框架思路**：
  - 从社交平台用户评论中创建多 LLM Agent，让它们就特定话题进行辩论
  - 结合 **GraphRAG** 进行知识检索，让辩论有数据支撑
  - Agent 人格来源于真实用户评论风格（非预设人设）
- **对本项目的参考价值**：
  - GraphRAG 结合辩论的思路：可以让 Agent 在讨论规则时引用历史对局数据作为论据
  - 从数据中生成人格：未来可根据游戏历史动态调整 Agent 性格

#### Argus AI Debate — 基于贝叶斯推理的事实验证辩论
- **链接**：https://github.com/Argus-Framework/argus-ai-debate
- **星标**：5 ⭐
- **框架思路**：
  - 多 Agent 通过辩论验证 claim 的真实性
  - 使用 **贝叶斯推理** 更新信念，**对抗性证据** 挑战结论
  - 输出 **校准判决（Calibrated Verdicts）**：不是简单的"对/错"，而是带置信度的判断
- **对本项目的参考价值**：
  - 贝叶斯信念更新：可用于 Agent 对规则提案的评估——初始倾向 + 辩论证据 → 最终投票
  - 校准判决：投票时不仅表达支持/反对，还附带置信度，便于分析

### 9.3 投票/辩论框架设计模式总结

基于以上调研，多智能体投票/辩论系统的通用设计模式如下：

```
提案阶段                  辩论阶段                    投票阶段
┌──────────┐       ┌─────────────────┐       ┌──────────────┐
│ 提案生成  │       │  结构化辩论      │       │  投票聚合     │
│ ────────  │       │  ──────────     │       │  ────────    │
│ • 格式校验│  ──→  │  • 正方/反方    │  ──→  │  • 多数投票   │
│ • 联署门槛│       │  • 裁判评估     │       │  • 加权投票   │
│ • 初审过滤│       │  • 多轮迭代     │       │  • 分层门槛   │
│ • 相似合并│       │  • 信念更新     │       │  • 纠错验证   │
└──────────┘       └─────────────────┘       └──────────────┘
```

**学术验证的关键结论**：

| 结论 | 来源 | 对本项目的影响 |
|------|------|--------------|
| 辩论+投票混合优于单一模式 | Debate or Vote (NeurIPS 2025) | 验证了当前"先辩论后投票"的设计 |
| 多人格设置影响辩论质量 | Debate or Vote | 验证了"性格化 AI 玩家"的价值 |
| 结构化辩论优于自由讨论 | MALLM | 可考虑给辩论加更多结构（如正反方时间分配） |
| RL 可优化辩论策略选择 | KylJin/Werewolf (NeurIPS 2024) | 未来可用 RL 训练讨论策略 |
| 信息不对称辩论更接近真实场景 | Sparse MAD | 可限制阵营间的信息可见性 |
| 超前K票投票比简单多数更稳健 | MAKER Framework | 可用于核心规则变更的高门槛投票 |

### 9.4 对当前系统的改进建议

基于调研，建议本项目规则讨论系统可考虑的优化方向：

1. **分层投票门槛**：小参数调整用简单多数，新增/删除规则用绝对多数，核心机制变更用超级多数
2. **结构化辩论**：明确正方（提案人）和反方（反对者）的发言配额，而非完全随机抽人
3. **信念显性化**：让 Agent 在辩论前后分别输出对提案的支持概率，观察辩论是否真正改变了立场
4. **多提案并行**：参考 MALLM 的讨论协议，同时对比讨论多个提案而非只投第一个
5. **历史数据引用**：辩论时让 Agent 引用 game_history.json 中的胜率/时长数据作为论据

## 十、Debate or Vote 深度分析

> 论文：*Debate or Vote: Which Yields Better Decisions in Multi-Agent Large Language Models?* (NeurIPS 2025 Spotlight)
> 作者：Hyeong Kyu Choi, Xiaojin Zhu, Sharon Li（UW-Madison）
> 源码：https://github.com/deeplearning-wisc/debate-or-vote

### 10.1 核心发现

该论文将多智能体辩论（MAD）拆解为两个独立组件：**多数投票（Majority Voting）** 和 **跨智能体辩论（Debate）**，并通过 7 个 NLP 基准实验验证：

| 结论 | 说明 |
|------|------|
| **多数投票贡献了 MAD 的大部分性能增益** | 简单的 N 个 agent 独立作答 + 投票汇总，已经能获得接近 MAD 的效果 |
| **辩论本身不改善期望正确率** | 理论证明辩论过程是 martingale（鞅过程），agent 信念轨迹的期望不变 |
| **偏向修正的干预可以提升辩论效果** | 当辩论更新被引导向正确方向时（biased belief update），性能才真正提升 |
| **异构人格设置影响辩论质量** | `--multi_persona` 给不同 agent 分配不同专家角色（数学家、律师等），可提升多样性 |

### 10.2 架构设计

```
┌─────────────────────────────────────────────────────────┐
│                      main.py (入口)                      │
│  - 加载数据集（arithmetics/gsm8k/hellaswag 等 7 个）      │
│  - 创建 N 个 agent（默认 5 个）                           │
│  - 循环 debate_rounds 轮                                 │
└──────────────────────┬──────────────────────────────────┘
                       ▼
         ┌─────────────────────────┐
         │   Round 0: 初始意见收集   │
         │   每个 agent 独立回答问题  │
         └────────────┬────────────┘
                      ▼
         ┌─────────────────────────┐
         │   Round 1~R: 辩论循环    │
         │  ┌───────────────────┐  │
         │  │ get_new_message() │  │
         │  │ 构造辩论上下文消息  │  │
         │  └────────┬──────────┘  │
         │           ▼             │
         │  ┌───────────────────┐  │
         │  │   engine() 推理    │  │
         │  │  批量生成新回答     │  │
         │  └────────┬──────────┘  │
         │           ▼             │
         │  ┌───────────────────┐  │
         │  │  evaluate() 评估   │  │

         │  └───────────────────┘  │

```

### 10.3 消息管理

Debate or Vote 的消息管理是**轻量级的纯字典传递**，没有使用消息总线或 MsgHub。

#### 初始意见收集（Round 0）

```python
# 同质化 agent：所有人收到相同的 user 消息
messages = [{"role": "user", "content": x + SUFFIX}] * args.num_agents

# 异构 agent：每人有不同的 system prompt（专家角色）
messages = []
for name, sys in personas.items():
    messages.append([
        {"role": "system", "content": sys},
        {"role": "user", "content": x + SUFFIX}
    ])

responses = engine(messages, agent, args.num_agents)
agent_responses = dict(zip(agent_names, responses))
```

#### 辩论消息构造（get_new_message）

这是整个框架的核心，有 3 种模式：

**1. 去中心化 MAD（Decentralized，默认）**

每个 agent 看到所有其他 agent 的最新回答：
```
"These are the recent opinions from other agents:
 Agent1's response: ...
 Agent2's response: ...
 This was your most recent opinion: ...
 Use these opinions carefully as additional advice to revise your
 recent opinion to give your final answer to the question: [原始问题]"
```

**2. 稀疏 MAD（Sparse，`--sparse`）**

每个 agent 只看到相邻两个 agent 的回答（环形拓扑）：
```python
peers = [agents[(i-1) % len(agents)], agents[(i+1) % len(agents)]]
```

**3. 中心化 MAD（Centralized，`--centralized`）**

只有中心 agent（Agent 0）看到所有人的意见，其他 agent 只看到中心 agent 的意见。

#### 消息不累积

**关键差异**：每轮辩论的消息是**无状态**的——不保留前几轮的对话历史，只包含当前轮的其他 agent 回答。这与 AgentScope 的 MsgHub（自动累积历史）形成对比。

### 10.4 投票/决策机制

答案提取和投票汇总在 `evaluator.py` 中实现：

```python
# 1. 从 LLM 输出中用正则提取答案
pred = re.findall(r"\{(.*?)\}", response)[-1]
pred = float(pred.replace("final answer:", "").strip())

# 2. 多数投票：Counter 统计 + 平局随机选
counter = collections.Counter([x for x in final_answers if x != ""])
max_count = max(counter.values())
most_common = [key for key, value in counter.items() if value == max_count]
debate_answer = random.choice(most_common)  # 平局随机选
```

| 评估方式 | 数据集 | 方法 |
|----------|--------|------|
| 精确匹配 | arithmetics, gsm8k | 正则提取 `{final answer: X}` |
| 选择题匹配 | hellaswag, pro_medicine, formal_logic, csqa | 正则提取 `{final answer: (X)}` |
| ROUGE 评分 | cnn_daily | 取 ROUGE-L 最高的摘要 |

### 10.5 模型推理引擎（engine）

```python
def engine(messages, agent, num_agents=1):
    # 批量 tokenize + padding
    prompts = [agent.tokenizer.apply_chat_template(msgs, ...) for msgs in messages]
    inputs = agent.tokenizer(prompts, return_tensors='pt', padding=True, truncation=True)
    # HuggingFace generate
    outputs = agent.huggingface_model.generate(
        input_ids, attention_mask,
        max_new_tokens=512, do_sample=True, temperature=1.0, top_p=0.9,
    )
    # 只解码新生成的 token
    for input_id, sequence in zip(input_ids, generated_sequences):
        gen_only = sequence[len(input_id):]
        decoded = agent.tokenizer.decode(gen_only, skip_special_tokens=True)
```

**特点**：所有 agent 共享同一个模型权重，通过 batch 推理实现"多个 agent"，差异化来自：
- 不同的 system prompt（`--multi_persona` 模式）
- 不同的辩论上下文（看到不同 agent 的回答）
- `temperature=1.0` 带来的随机性

## 十一、AgentScope 多智能体辩论示例深度分析

> 源码地址：https://github.com/agentscope-ai/agentscope/tree/main/examples/workflows/multiagent_debate
> 参考论文：[EMNLP 2024](https://aclanthology.org/2024.emnlp-main.992/)

### 11.1 整体架构

这是一个极简的"两辩手 + 一裁判"辩论框架：

```
┌─────────────────────────────────────────┐
│              Main Loop                   │
│                                          │
│  while True:                             │
│    ┌─ MsgHub([Alice, Bob, Moderator]) ─┐│
│    │  Alice 发表正方观点                 ││
│    │  Bob 发表反方观点                   ││
│    │  (消息自动广播给所有参与者)          ││
│    └────────────────────────────────────┘│
│    Moderator 在 MsgHub 外判定            │
│    → finished=True → 结束               │
│    → finished=False → 继续下一轮         │
└─────────────────────────────────────────┘
```

### 11.2 消息管理

```python
async with MsgHub(participants=[alice, bob, moderator]):
    await alice(
        Msg("user", "You are affirmative side, Please express your viewpoints.", "user"),
    )
    await bob(
        Msg("user", "You are negative side. You disagree with the affirmative side.", "user"),
    )
```

**关键设计**：
- Alice 和 Bob 在同一个 MsgHub 内，**消息自动广播**——Alice 说完后，Bob 自动能看到 Alice 的发言
- MsgHub 内的消息**自动累积到每个参与者的记忆**中，形成完整的辩论历史
- Moderator 也在 MsgHub 的 participants 中，所以也能看到双方的辩论内容

### 11.3 裁判在 MsgHub 外判定

```python
# Moderator 在 MsgHub 外部调用 → Alice 和 Bob 看不到裁判的判定
msg_judge = await moderator(
    Msg("user", "...have the debate finished?...", "user"),
    structured_model=JudgeModel,
)
if msg_judge.metadata.get("finished"):
    print("The correct answer is:", msg_judge.metadata.get("correct_answer"))
    break
```

**设计意图**：裁判的判定不应影响辩手的论点，所以放在 MsgHub 外。辩手只看到彼此的发言，不知道裁判怎么想。

### 11.4 结构化输出

```python
class JudgeModel(BaseModel):
    finished: bool = Field(description="Whether the debate is finished.")
    correct_answer: str | None = Field(
        description="The correct answer, only if finished. Otherwise None.",
        default=None,
    )
```

裁判每轮输出两个字段：是否结束 + 正确答案。用 Pydantic 保证格式合法。

### 11.5 提示词设计

**辩手（Solver）系统提示词**：
```
You're a debater named {name}. Hello and welcome to the debate competition.
It's not necessary to fully agree with each other's perspectives, as our
objective is to find the correct answer.
The debate topic is: {topic}. Use Chinese to answer the question.
```

**裁判（Aggregator）系统提示词**：
```
You're a moderator. There will be two debaters involved in a debate competition.
They will present their answer and discuss their perspectives on the topic:
{topic}
At the end of each round, you will evaluate both sides' answers and decide
which one is correct.
```

**特点**：
- 提示词鼓励**不必完全同意**，保持独立思考
- 裁判的角色是判定正确答案，而非调和分歧
- 使用 `DashScopeMultiAgentFormatter` 处理多 agent 对话格式

### 11.6 与 Debate or Vote 的对比

| 维度 | AgentScope 辩论 | Debate or Vote |
|------|-----------------|----------------|
| 消息累积 | ✅ MsgHub 自动累积全部历史 | ❌ 无状态，每轮只传最新意见 |
| Agent 数量 | 2 辩手 + 1 裁判 | N 个同质/异质 agent |
| 终止条件 | 裁判判定 `finished=True` | 固定 `debate_rounds` 轮 |
| 决策方式 | 裁判判定正确答案 | 多数投票 |
| 信息拓扑 | 全可见（MsgHub 广播） | 可配置（全/稀疏/中心化） |
| 角色分化 | 正方 vs 反方 | 同质或按专业领域分 |
| 记忆持久化 | 支持（ReActAgent 内置） | 不支持 |

### 11.7 对本项目的启示

| AgentScope 辩论特性 | 本项目可借鉴点 |
|---------------------|---------------|
| MsgHub 自动广播 + 历史累积 | 辩论阶段使用 auto_broadcast，让所有人看到彼此发言 |
| 裁判在 MsgHub 外部判定 | 主持人评估提案可在辩论 hub 外部进行，不影响辩论进程 |
| 结构化终止判定（JudgeModel） | 可用结构化输出让主持人判定"辩论是否充分" |
| 正方/反方角色分化 | 规则辩论可明确提案支持者和反对者的角色 |
| `DashScopeMultiAgentFormatter` | 多 agent 对话需要专用的消息格式化器 |

## 十二、三个项目的综合对比与设计建议

### 12.1 架构模式对比

| 维度 | AgentScope 狼人杀 | Debate or Vote | AgentScope 辩论 |
|------|-------------------|----------------|-----------------|

| **消息管理** | MsgHub 嵌套 + 动态广播控制 | 无状态字典传递 | MsgHub 自动广播 |
| **流程控制** | sequential + fanout pipeline | for 循环 + 批量推理 | while + MsgHub |
| **结构化输出** | Pydantic多模型 + 动态Literal | 正则提取 | Pydantic 单模型 |
| **记忆管理** | ReActAgent + JSONSession | 无 | ReActAgent 内置 |
| **决策方式** | majority_vote | majority_vote + 正则 | 裁判判定 |
| **角色差异化** | 角色提示词 + 结构化约束 | multi_persona 系统prompt | 正方/反方 |

### 12.2 对本项目规则讨论系统的综合建议

基于三个项目的深度分析，建议以下架构：

```
规则讨论阶段架构建议
━━━━━━━━━━━━━━━━━━━━

1. 消息管理
   - 讨论阶段：MsgHub 模式，auto_broadcast=True，所有人可见
   - 投票阶段：fanout 模式，auto_broadcast=False，独立投票后统一公布
   - 主持人评估：在 MsgHub 外部进行，不干扰讨论

2. 流程控制
   - 提案阶段：fanout_pipeline 并行收集提案
   - 辩论阶段：sequential_pipeline 轮流发言（或按正/反方交替）
   - 投票阶段：fanout_pipeline 并行投票
   - 终止判定：结构化输出（finished: bool）由主持人判定

3. 结构化输出
   - 投票：Pydantic + Literal 约束合法选项
   - 提案：结构化字段（rule_key, new_value, reason）
   - 辩论：可选 reach_agreement 标记提前结束

4. 记忆与历史
   - Agent 内置记忆累积辩论历史
   - 跨局持久化记忆（JSONSession 模式）
   - 反思机制：每次投票后 agent 反思自己的判断

5. 辩论有效性
   - 参考 Debate or Vote 的发现：引用 game_history 数据作为"偏向修正"
   - 避免纯自由辩论（martingale → 期望不变）
   - 结构化辩论：先正方陈述理由，再反方质疑，再自由讨论
```
